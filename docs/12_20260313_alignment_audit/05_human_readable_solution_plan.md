# 面向业务理解的方案定义 v1

这份文档不是“当前代码怎么写”，而是“这个项目从业务上应该怎么定义”。

目标是先把需求钉死，让后面的 SQL、摘要表、标签系统和 UI 都围绕同一套逻辑工作，而不是随着实现迭代漂移。

## 1. 项目核心目标

这个项目不是单纯做 IP 分类，而是做三件事：

1. 把原始 IP 数据整理成稳定、可解释的网络对象。
2. 让相同网络特性的数据尽量汇聚，让不同网络特性的数据尽量分开。
3. 为后续画像分析和 UI 可视化提供更纯净、更稳定、更可解释的数据底座。

换句话说：

- 前面的分块、切分、分类，是为了“纯化对象”
- 后面的画像、标签、展示，是为了“理解对象”

如果前面的对象不纯，后面的标签和 UI 一定会越来越乱。

## 2. 三个库的中文命名

为了避免 `H/E/F` 这种迭代代号影响理解，先给出面向业务的中文名字。

### H 库：核心连续网络库

含义：

- 由连续 IP 组成
- 经过切分后，内部网络行为相对一致
- 属于核心网络对象

你可以把它理解成：

- 真正值得做“网络块画像”的主库

### E 库：稀疏聚合网络库

含义：

- 没能进入 H
- 但在更粗粒度上，仍然表现出局部聚集性
- 适合用 `/27` 原子和连续原子段来聚合

你可以把它理解成：

- 不够连续到进入 H，但又不是完全零散的网络对象

### F 库：零散剩余 IP 库

含义：

- 没能进入 H
- 也没能进入 E
- 最终剩余的零散 IP

你可以把它理解成：

- 兜底集合
- 仍然可以做画像，但不应该伪装成“成段网络”

## 3. 整体方案，用人话讲

整个流程可以理解成下面 10 步。

### 第 1 步：接收原始数据

目的：

- 把项目的原始输入固定下来

怎么做：

- 读取原始 IP 数据
- 保留后续可能需要做画像的原始字段

输出对象：

- 原始 IP 数据集

### 第 2 步：整理成统一的 IP 成员表

目的：

- 把所有数据整理成“每个 IP 一行”的统一结构
- 让后面的连续性判断、异常标记、分块、画像都基于同一层输入

怎么做：

- 以 IP 为最小粒度
- 统一基础字段命名
- 补齐定位字段和基础统计字段

输出对象：

- 统一 IP 成员表

这一步的本质：

- 它不是分类结果
- 它是整个系统的统一起点

### 第 3 步：标记异常 IP

目的：

- 给异常 IP 打标签
- 防止异常值污染后面的统计、切分和标签结果

怎么做：

- 将异常 IP 表关联到统一 IP 成员表
- 为每个 IP 增加异常标记

当前按你的设计，推荐原则是：

- 异常 IP 仍然保留在数据里，方便审计和追溯
- 但异常 IP 不参与后续的关键统计计算

这一步在业务上非常关键，因为异常 IP 会直接影响：

- 平均值
- 密度
- Wi-Fi / 移动占比
- 切分阈值
- 标签判断

所以你的设计里这一点是合理的。

### 第 4 步：识别连续地址段，形成 H 的基础对象

目的：

- 找出哪些 IP 在地址上是连续的
- 形成最底层的连续地址段对象

怎么做：

- 先按 IP 连续性形成连续段
- 这个阶段的核心是“地址连续”，不是“画像一致”

注意：

- 这里不是最终 H 库
- 这里只是在建立 H 库的原始基础段

### 第 5 步：对连续段进行切分，纯化网络对象

目的：

- 把一个地址上连续、但属性上混杂的段拆开
- 让同一个块内部的网络特征尽量一致

怎么做：

- 根据运营商变化切分
- 根据均值变化切分
- 根据 Wi-Fi 变化切分
- 根据前后段差异切分
- 根据更细粒度窗口做补充切分
- 对超长段做强制切分

这一步的本质：

- 不是为了切得更碎
- 而是为了让块内部更纯

所以切分逻辑应该遵守这个原则：

- 只要新增切分方法，就必须提升对象纯度或解释力
- 不能只是为了速度方便
- 更不能因为实现方式变化，破坏前后流程逻辑

### 第 6 步：对切分后的块做网络评估，决定是否进入 H

目的：

- 把不够核心的块排除出 H
- 保证 H 只保留真正值得作为主网络对象分析的块

怎么做：

- 对切分后的块做规模和密度评估
- 将微型网络和小型网络排除出 H
- 同时增加最小块大小门槛，避免单 IP 或极小块误入 H

推荐理解：

- H 不是“所有连续块”
- H 是“经过切分和评估后，仍然足够像核心网络块的连续块”

### 第 7 步：对 H 库做块级画像

目的：

- 对核心连续网络对象做系统化描述

怎么做：

- 统计 H 块内的行为特征
- 生成可用于 UI 展示和标签分析的摘要层

这一步适合做：

- 网络类型结构
- 时段结构
- 设备结构
- 风险结构
- 运营商结构

### 第 8 步：对未进入 H 的对象，构建 E 库

目的：

- 从未进入 H 的剩余 IP 里，再找出“虽然不够进入 H，但仍然有局部聚集性”的对象

怎么做：

- 以更粗的 `/27` 为原子
- 对原子做密度判断
- 将连续的原子拼成段
- 把满足条件的对象纳入 E

这一步的核心思想：

- H 解决“连续核心网络”
- E 解决“非 H，但仍然能聚起来的网络”

这里最关键的规则必须写死：

- 连续原子段最短长度到底是多少
- 是 `2` 还是 `3`

这个规则不写死，E/F 边界就永远不稳定。

### 第 9 步：对 E 库做段级画像

目的：

- 对 E 这种“稀疏但有聚集性”的对象做独立画像

怎么做：

- 按段级做统计
- 提供给标签系统和可视化系统

注意：

- E 的画像口径不一定和 H 完全相同
- 但字段体系应尽量兼容，方便横向对比

### 第 10 步：将剩余对象归入 F，并做 F 画像

目的：

- 接住所有既不属于 H，也不属于 E 的 IP
- 让项目在分类上闭环

怎么做：

- 对剩余 IP 直接进入 F
- 对 F 做单 IP 或轻聚合画像

注意：

- F 不是失败数据
- F 是剩余零散对象库
- 它仍然值得研究，只是研究方式不能和 H 完全一样

## 4. 这套流程里，异常 IP 应该怎么处理

根据你刚才的设计，我建议把规则写成下面这句，这样最清楚：

> 异常 IP 保留用于审计和追溯，但不参与连续段统计、切分信号计算、画像统计和标签统计。

这句话的重要性非常高，因为它直接决定：

- 连续段的统计是否会被异常值带偏
- 切分是否会被异常波动误触发
- H/E/F 的边界是否稳定
- 标签是否会误判

如果后面你确认“异常 IP 不打断连续性”，那这条规则会更完整：

> 异常 IP 不打断地址连续性，但不参与所有关键统计与判断。

这是目前最符合你设计意图的表达。

## 5. 字段体系，不按表讲，按用途讲

你刚才提到一个非常重要的点：

- 最后一步画像需要很多字段
- 前面的处理流程，本质上是为了让这些字段的特征更突出、更纯净

这个判断是对的。

所以字段体系不应该按“哪张表有什么列”来定义，而应该按“这些字段在流程中起什么作用”来定义。

建议把字段分成 6 类。

### A. 身份与定位字段

作用：

- 识别一个 IP、一个段、一个块是谁

典型字段：

- IP
- IP 整型值
- run_id
- shard_id
- 起始 IP
- 结束 IP
- 原子编号
- 块编号
- 段编号

### B. 异常控制字段

作用：

- 告诉系统哪些数据不能参与关键计算

典型字段：

- 是否异常
- 异常来源
- 异常规则命中数
- 是否有效

### C. 切分信号字段

作用：

- 决定一个连续段是否需要被切开

典型字段：

- 运营商
- 平均设备数
- 平均上报数
- Wi-Fi 设备数或比例
- 移动设备数或比例
- VPN 设备数或比例
- 左右窗口差异值
- 密度变化值

### D. 分类判断字段

作用：

- 决定对象进入 H、E 还是 F

典型字段：

- 连续段长度
- 有效 IP 数
- 密度
- 评分
- 网络等级
- `/27` 原子密度
- 连续原子段长度

### E. 画像分析字段

作用：

- 给 UI、标签系统和人工分析提供可解释的对象描述

典型字段：

- 总设备数
- 总上报数
- Wi-Fi 设备数
- 移动设备数
- VPN 设备数
- Root 上报
- 代理上报
- ADB 上报
- 工作时上报
- 工作日上报
- 周末上报
- 深夜上报
- 活跃天数
- 制造商数
- 型号数
- SSID/BSSID
- 运营商分布

### F. 展示与标签派生字段

作用：

- 让 UI 更可视化，让标签系统更稳定

典型字段：

- 各类比例字段
- 平均值字段
- 排名前运营商
- 主网络类型
- 异常占比
- 不稳定 IP 占比
- 标签命中结果

## 6. 关于切分策略的设计原则

你提到“现在有很多切分方法，未来还会增加”，这个判断是对的。

所以切分不应该被写成一堆临时规则，而应该被定义成“可扩展的纯化机制”。

建议固定 4 条原则。

### 原则 1：新增切分是为了提升纯度，不是为了追求步骤复杂

新增切分的理由必须是：

- 让块内部更一致
- 让后续画像更清晰
- 让标签更稳定

如果一个切分只是让实现更快，但破坏了对象语义，就不应该直接进入正式逻辑。

### 原则 2：性能优化不能偷偷改变业务口径

如果为了速度改了：

- 窗口大小
- 边界定义
- 参与统计的字段
- 异常值处理方式

那这已经不是单纯优化，而是改规则。

此时必须：

- 版本化
- 做前后对比
- 明确说明影响了哪些库

### 原则 3：新增切分后，必须检查对 H/E/F 边界的影响

每增加一个切分策略，都要检查：

- H 块数是否变化
- H 成员数是否变化
- E/F 分流是否变化
- 标签分布是否变化
- 对象纯度是否真的变好

不能只看“跑通了”。

### 原则 4：切分层和画像层必须分开

切分层回答的是：

- 这个对象该不该拆

画像层回答的是：

- 这个对象是什么样

如果把两者混在一起，后续会越来越难解释。

## 7. UI 设计应该服务什么

后续 UI 不是用来“补救逻辑不清”，而是用来“放大已经清楚的逻辑”。

因此 UI 最适合做的，是把每个对象的来源和特征可视化。

建议 UI 围绕下面几类信息设计：

### 1. 对象是什么

- 是 H、E 还是 F
- 中文名称是什么
- 为什么进入这个库

### 2. 它是怎么形成的

- 原始连续段是什么
- 中间切了几次
- 为什么切
- 最终为什么进入当前库

### 3. 它长什么样

- 设备结构
- 上报结构
- 时段结构
- 风险结构
- 运营商结构

### 4. 它被打了什么标签

- 主标签是什么
- 附加特征标签有哪些
- 哪些条件命中的

## 8. 在方案层，必须先钉死的 7 个问题

这 7 个问题不定，后面实现一定继续漂：

1. H 库的中文定义是什么
2. H 的最小块大小是否固定为 `>=4`
3. 全异常块是否允许进入业务 H/E/F
4. 异常 IP 是否打断连续性
5. E 的最小连续原子段长度是多少
6. 标签是唯一主标签，还是主标签 + 附加标签
7. 摘要层是否必须完整镜像分类层

## 9. 我对你设计的总体评价

你的设计方向是对的，核心思路可以总结成一句话：

> 先把网络对象切纯，再做分类，再做画像，再做可视化。

这是正确的顺序。

当前真正需要补强的，不是大框架，而是 3 个地方：

1. 要把“异常 IP 如何参与计算”写成硬规则。
2. 要把“H/E/F 的边界条件”写成硬规则。
3. 要把“新增切分方法的准入原则”写成硬规则。

只要这三部分钉住，后面的实现就能稳定很多。
