# 异常清单

## A1. `D库` 概念未落地，当前技术实体只有 `H/E/F`

问题：

- 你口中的 `D库` 在当前代码、表、API、配置中没有明确实体。

我怎么理解：

- 这很可能是业务口径与技术口径没有重新对齐。
- 如果不先定义 `D库 = 什么`，后面所有“它不该含单 IP”“它和其他标签什么关系”的讨论都会失真。

证据：

- 当前 schema 下无 `d_*` 主表。
- 当前主链路表是 `h_* / e_* / f_*`。
- 当前 API 也只围绕 `H/E/F` 展开。

影响：

- 业务对的是 `D`
- 代码跑的是 `H`
- 页面展示的是 `H summary`
- 三层讨论对象可能根本不是一个东西

选项：

- A. `D库 = H库`
  说明：最省改动，后续只需要统一命名和口径。
- B. `D库 = H库` 的子集
  说明：例如只保留“大型/超大网络”，那就需要新增显式过滤层。
- C. `D库` 是独立第四库
  说明：需要新增明确入库规则、表、摘要和 UI，不建议在未定规则前直接编码。

## A2. 小碎块问题没有被根修，只是被 H 准入门槛挡住

问题：

- 最新 `sg_004` 的 H 已不含单 IP，但 `profile_final` 里仍然存在大量 `<4 IP` 且高 tier 的块。

我怎么理解：

- 真正的问题仍在 Step 04 切分和 Step 04P 评分阶段。
- Step 05 只是最后加了一个 `member_cnt_total >= 4` 门槛。

证据：

- `sg_004` 中 `h_blocks` 的 `<4 IP` 为 `0`
- 但 `profile_final` 中 `<4 IP` 且 `network_tier_final` 属于 H 档的块有 `7,530`
- `sg_001` 历史 run 里，H 仍有 `5,951` 个单 IP 块和 `1,565` 个 `2-3 IP` 块

影响：

- 任何直接读 `profile_final` 的分析都还会看到这些异常小块
- 如果以后改回 H 准入条件，问题会立刻复发
- 用户看到“库里已经修好”，实际上只是下游门卫在拦

选项：

- A. 维持现状，只在 H 准入层挡住 `<4 IP`
  说明：改动最小，但问题继续留在上游。
- B. 在 Step 04/04P 引入“业务最小块尺寸”规则
  说明：从源头减少异常碎块，后续最稳定。
- C. 保留原始碎块，同时再产出一层“业务块”
  说明：适合既要审计原貌又要稳定分类的场景，但实现复杂。

## A3. `E` 当前把短 run 也算进 E，和文档口径冲突

问题：

- 当前 SQL 中，`run_len < 3` 的短 run 只是被标记为 `short_run`，并没有被排除出 `E`。

我怎么理解：

- 现在系统实际上把“短 run”当成 E 的正式成员。
- 这和多份文档中“`run_len >= 3` 才进 E，否则进 F”的描述冲突。

证据：

- `sg_004` 中：
  - `short_run_cnt = 62,352`
  - `short_run_member_cnt = 937,106`
- 也就是说，接近 `94 万` IP 的归属，取决于你到底要不要把短 run 放进 E

影响：

- E/F 规模、标签画像、业务解读都会变
- 如果你认为 `D/H` 是大网络库，而 E 应该只承接“足够稳定的非连续段”，那短 run 继续进 E 会让 E 边界变宽

选项：

- A. 严格按文档执行，`run_len < 3` 不进 E，回流 F
  说明：边界清晰，最符合现有文档。
- B. 保持现实现状，短 run 继续进 E，但在文档/UI 明确说明
  说明：不改数据分布，只修口径。
- C. 短 run 单独成库或单独标签层
  说明：最利于审计，但会引入新的业务对象。

## A4. H 摘要层不是 H 分类层的完整镜像

问题：

- `h_block_summary` 当前缺少一部分真实 H 块。

我怎么理解：

- 页面看到的是“H 摘要”，不是“完整 H”。
- 缺的这部分刚好是 `valid_cnt=0` 的 H 块。

证据：

- `sg_004` 中：
  - `h_blocks = 21,484`
  - `h_block_summary = 21,323`
  - 缺口 `161`
- 这 `161` 个块全部满足：
  - `valid_cnt = 0`
  - 汇总容量 `10,889` IP
- `build_h_block_summary.py` 会先排除异常 IP，再基于聚合结果做内连接

影响：

- 分类层、摘要层、页面层不是同一口径
- 审计人员容易误以为“页面没看到 = 库里没有”
- 标签分析天然漏掉这批块

选项：

- A. 让 `h_block_summary` 完整镜像 `h_blocks`
  说明：即使 `valid_cnt=0`，也保留一行摘要，只把画像字段置空或标记。
- B. 保持当前做法，但明确重命名为“valid-only H summary”
  说明：减少误解，但前端和文档都要同步改名。
- C. 双轨输出
  说明：一个完整 summary，一个 valid-only summary，适合严格审计场景。

## A5. 最新 `sg_004` 没有 QA 验收记录

问题：

- 最新 run 虽然数据已经分到 H/E/F，但 `qa_assert` 表里没有 `sg_004` 的记录。

我怎么理解：

- 当前你看到的是一个“有结果但没有正式验收证据”的 run。
- 这会让后续每一次口径讨论都少了一层可追溯证据。

证据：

- `qa_assert` 当前只有 `sg_001`
- `sg_004` 没有 QA 记录

影响：

- 现在只能通过人工 SQL 重查来证明守恒和互斥
- 后续如果切换 run，页面和数据库可能默认指向一个未验收版本

选项：

- A. 没有 QA 记录的 run 不允许作为正式分析口径
  说明：最严格，证据链最完整。
- B. 允许作为预览 run，但页面必须明确标注“未验收”
  说明：利于迭代，但不能当正式结果。
- C. 改造 QA 方式，不再依赖 `qa_assert`，改为页面实时校验
  说明：不建议优先走，复杂度高。

## A6. `ALL_ABNORMAL_BLOCK` 现在只是标签，不再是真正剔除

问题：

- 当前工作区里，全异常块被标记但不删除；历史 run 则还保留过真实 drop。

我怎么理解：

- 这不是一个“纯技术细节”，而是核心业务选择。
- 它决定异常 IP/异常块是否继续参与 H/E/F 分类和标签分析。

证据：

- 当前 Step 03 `keep_flag = true`
- `sg_004` 中 `keep_members = source_members`
- `sg_001` 中仍能看到旧的 `drop_blocks`

影响：

- 同一个项目里，不同 run 的“Keep”含义已经变了
- 历史对比会失真
- 后续“D/H 不该含什么”会被这个选择直接影响

选项：

- A. 全异常块只标记不删除
  说明：适合审计原貌，但下游必须自己决定是否接纳。
- B. 全异常块完全退出主分类链路
  说明：主库更干净，但失去完整审计视图。
- C. 双轨保留
  说明：保留原始链路，同时在业务链路里剔除。

## A7. 当前标签系统是“漏斗排他”，不是“多标签并存”

问题：

- 你提到“多个标签的问题”，而当前系统默认一个对象只会命中漏斗中的第一个主标签。

我怎么理解：

- 当前标签更像“主类目”。
- 如果你期待一个对象同时拥有多个业务标签，现在的实现方向并不支持。

证据：

- H/E/F 的 profiling 标签都是按顺序执行，并从剩余池继续筛选
- 前一标签命中后，后续标签不再继续命中

影响：

- 多标签分析会天然丢失交叉语义
- 标签顺序的变化会直接改变结果
- 当前“标签问题”本质上可能不是阈值问题，而是标签模型选错了

选项：

- A. 维持“唯一主标签”，其他特征改为附加属性
  说明：最稳定，也最适合当前 UI。
- B. 改成多标签命中模型
  说明：表达力更强，但 UI、API、统计口径都要重写。
- C. 双层标签
  说明：保留主标签漏斗，同时增加多值特征标签层。

## A8. UI 仍在把 H 当成“仅中型网络”，和当前 H 定义不一致

问题：

- 当前部分页面和研究接口仍按“`H = 中型网络`”来消费数据。

我怎么理解：

- 数据库里现在的 H 已经是“中型 + 大型 + 超大，且块大小 >= 4”。
- 但页面有些地方仍把 H 当成旧口径子集。

证据：

- Dashboard 文案仍写 `H 类 (中型网络)`
- `research.py` 中部分 H 相关查询仍显式限制 `network_tier_final = '中型网络'`

影响：

- 页面展示的 H 不是数据库里的完整 H
- 用户会误以为 H 规模、分布、画像都比真实值更小
- 后续即使数据库口径修好了，页面仍会继续误导

选项：

- A. 页面全部改成当前 H 真口径
  说明：最直接，避免继续误导。
- B. 页面明确拆成“完整 H”和“中型 H 子集”
  说明：适合业务确实想同时保留两个观察视角。
- C. 数据库回退到“仅中型网络”的 H 定义
  说明：不建议先走，牵涉面太大，而且与当前数据现状不一致。
