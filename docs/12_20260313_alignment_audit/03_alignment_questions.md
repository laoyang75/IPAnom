# 需要你先拍板的问题

下面这些问题不先定，后面的代码修复会反复打架。

## Q1. `D库` 到底指什么

推荐先选：

- A. `D库 = 当前 H库`
- B. `D库 = 当前 H库` 的子集
- C. `D库` 是独立新库

我当前倾向：

- A 或 B

原因：

- 现在技术上根本没有单独的 `D库` 实体。
- 如果直接按 C 走，会让审计、修复、UI、数据库都同时扩容。

选择：A

## Q2. 大网络库是否允许单 IP / 小于 4 IP 的块

推荐先选：

- A. 不允许，`member_cnt_total >= 4` 是硬规则
- B. 允许，但要特殊标签化
- C. 允许，并直接当正常大网络处理

我当前倾向：

- A

原因：

- 这最符合你现在提出的核心诉求。
- 同时也和最新 `sg_004` 的 H 表现一致。

选择：A

## Q3. 大网络库是否允许 `valid_cnt = 0` 的全异常块

推荐先选：

- A. 不允许，必须 `valid_cnt > 0`
- B. 允许，但只放在审计视图，不进入业务摘要/UI
- C. 允许，并和正常块一样进入业务库

我当前倾向：

- B

原因：

- 当前 `sg_004` 里仍有 `161` 个 `valid_cnt=0` 的 H 块。
- 如果直接选 C，业务解释会很难。
- 如果直接选 A，又会丢失你可能需要的异常审计链路。

选择：B
## Q3A. 异常 IP 如何参与连续性、切分和标签统计

推荐先选：

- A. 异常 IP 不打断连续性，但不参与分块统计、切分信号、画像统计和标签统计
- B. 异常 IP 既不参与统计，也直接打断连续性
- C. 异常 IP 只做展示标签，不影响主链路计算

我当前倾向：

- A

原因：

- 这和你刚才描述的设计最一致。
- 一个连续段里混入异常 IP 时，地址段本身仍然连续，但所有统计和切分信号都只看 valid IP，避免异常值把均值、Wi-Fi 变化、切分判断带偏。
- 如果直接选 B，整个 H 基础分块逻辑会明显改变，影响会比“剔除异常值参与计算”大很多。

选择：A 但要注意，可能会在边界连续出现异常，导致边界切分出现问题，所以出现连续异常要特殊处理，这个之前应该有过程序跑补丁。

## Q4. `E` 是否必须要求 `run_len >= 3`

推荐先选：

- A. 必须，短 run 退回 F
- B. 不必须，短 run 继续进 E，但必须明文定义
- C. 短 run 独立成第四种中间对象

我当前倾向：

- A

原因：

- 这和现有文档最一致。
- 现在 `sg_004` 有 `937,106` IP 正受这个口径影响，必须尽快冻结。

选择：A

## Q5. 标签模型要“唯一主标签”还是“多标签并存”

推荐先选：

- A. 唯一主标签，附加特征另算
- B. 多标签并存
- C. 双层模型：主标签唯一，特征标签可多值

我当前倾向：

- C

原因：

- 现在的漏斗式标签很适合主标签。
- 但你提到的“多个标签问题”说明仅靠单标签不够。
- 双层模型能在不推翻现有 UI 的前提下扩展。

选择：C

## Q6. 摘要层是否必须完整镜像分类层

推荐先选：

- A. 必须，摘要层必须覆盖全部分类对象
- B. 不必须，但页面必须明确它是过滤后的视图
- C. 双轨摘要

我当前倾向：

- A

原因：

- 现在 `h_blocks != h_block_summary`，已经直接造成理解偏差。
- 先做到镜像，再决定页面如何筛。

选择：A

## Q7. 没有 QA 终验收记录的 run 能不能作为当前默认 run

这里的 `QA` 指的是：

- H/E/F 是否互斥
- Keep 是否守恒等于 H+E+F
- 是否存在幽灵 IP
- shard 输出是否完整

也就是“这次运行结果有没有被正式验收过”。

推荐先选：

- A. 不能
- B. 可以，但只能预览
- C. 可以，人工复核即可

我当前倾向：

- B

原因：

- `sg_004` 已有实际数据价值，但证据链不完整。
- 直接当正式口径风险太大。

选择：B
