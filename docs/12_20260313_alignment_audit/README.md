# 2026-03-13 数据审计对齐包

这是一份独立于现有实现的审计目录，目标是先把当前项目的真实数据逻辑、口径冲突和需要业务拍板的问题拆开说明，再决定后续代码怎么修。

本目录基于两类事实源：

1. 当前工作区中的 SQL / Python / WebUI 实现
2. 数据库 `rb20_v2_5` 中实际存在的运行结果，重点核对 `rb20v2_20260202_191900_sg_004`

注意：

- 当前工作区是脏树，SQL 与 WebUI 有未提交改动。
- 数据库中的历史 run 和当前工作区代码并不完全同版，必须区分“当前代码口径”和“历史数据口径”。
- 目前代码和数据库中没有找到明确的 `D库` 技术实体，当前主分类链路是 `H / E / F`。

文件说明：

- `01_data_logic_truth.md`: 当前数据处理逻辑真相，逐步说明每一步做什么、目的是什么、表之间如何衔接。
- `02_anomaly_register.md`: 已识别异常清单，附证据、影响和 A/B/C 处理选项。
- `03_alignment_questions.md`: 需要你拍板的核心问题，便于先对齐大框架。
- `04_code_fix_assessment.md`: 在未改代码前提下，对当前数据代码问题和修复顺序的初步评估。
- `05_human_readable_solution_plan.md`: 基于业务理解重新整理的方案定义，用中文名称、中文步骤和字段分层来钉需求。
- `06_business_solution_from_code.md`: 基于真实实现回填后的业务方案版，按“统一 -> 识别 -> 纯化 -> 分流 -> 画像/标签”展开。
- `07_implementation_spec_from_code.md`: 面向实现约束的规范定义版，强调分层、规则、参数与防偏移。
- `08_code_evidence_index.md`: 代码依据索引，标注直接证据与合理推断来源。
