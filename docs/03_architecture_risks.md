# 架构风险与脆弱性报告

> 对 RB20 v2.5 Pipeline 的工程架构、执行稳定性和可维护性进行风险评估。

## 1. 风险矩阵总览

| 风险 | 严重度 | 可能性 | 当前缓解措施 | 建议 |
|------|--------|--------|-------------|------|
| Shard 数据倾斜 | 🔴 高 | 🟡 中 | bucket 方案 | 完善 LPT 装箱 |
| 长链路中断恢复 | 🔴 高 | 🟡 中 | 幂等重跑 | 增加检查点 |
| 宽表 JOIN 内存爆破 | 🔴 高 | 🟡 中 | slim 表 | 持续优化 |
| QA 断言覆盖盲区 | 🟡 中 | 🟢 低 | 11 条 STOP | 补充统计类断言 |
| 无增量更新能力 | 🟡 中 | 🔴 高 | 无 | 设计增量方案 |
| 单点 PostgreSQL | 🟡 中 | 🟢 低 | 无 | 考虑读从库 |

## 2. Shard 数据倾斜

### 2.1 问题描述

ShardPlan 使用 `NTILE(64)` 分位切分，理论上均衡。但实际 IP 地址分布存在"成簇热区"现象——某些运营商的 IP 段高度密集，导致少数 Shard 的 `source_members` 行数远超平均值。

### 2.2 已有缓解: bucket 方案

`wenti.md` 已提出三阶段方案（评估→执行→收敛），核心是在 Shard 内按 `block_id_natural` 做二级 bucket 切片。

**效果评估**:
- ✅ 解决了 Step03（最耗时阶段）的长尾问题
- ✅ staging 表 + 后建索引减少了 WAL 和锁争用
- 🟡 但整体 Pipeline 其他阶段（04/04P/06/07/08）未受益

### 2.3 建议

1. 将 bucket 方案推广到所有 per-shard 阶段
2. 增加 ShardPlan 生成后的"倾斜度自检"（`max_rows/avg_rows > 3` 即告警）
3. 长期考虑动态 shard_cnt（当前硬编码 64）

---

## 3. 长链路容错与回滚

### 3.1 当前机制

- **幂等重跑**: 每步执行前清理 `run_id + shard_id` 旧数据
- **STOP 断言**: Phase 99 全局校验
- **run_id 隔离**: 不同批次完全隔离

### 3.2 脆弱点

| 场景 | 问题 | 影响 |
|------|------|------|
| Phase 04 某 Shard 超时 | 需要手动定位失败 Shard 并重跑 | 无自动恢复 |
| Phase 05 全局 H 提取后才发现数据错误 | 需要回滚 H + 重跑下游 R1/E/F | 连锁效应大 |
| Pipeline 执行中数据库重启 | 所有进行中的 Shard 任务中断 | 需要全量检查哪些完成、哪些需重启 |
| ShardPlan 需要调整 | 必须新 run_id 重跑全链路 | 浪费已完成的计算 |

### 3.3 建议

1. **检查点机制**: 给每个 Phase × Shard 维护 `step_status`（PENDING/RUNNING/DONE/FAILED），支持断点续跑
2. **编排器增强**: `orchestrate_fresh_start_v2.py` 增加失败 Shard 自动重试（最多 3 次）
3. **Phase 级回滚**: 设计"从 Phase N 重跑"能力，自动清理 Phase N 及其下游数据

## 4. 宽表 JOIN 性能

### 4.1 问题本质

`source_members` 有 100+ 列但每步只用 10~15 列。并发 JOIN 时：
- 读放大：每行读取 100+ 列但只用少数
- 内存压力：work_mem × 并发数 × 行大小
- temp 溢写：超出 work_mem 后溢出到磁盘

### 4.2 已有缓解

`wenti.md` 提出 `source_members_slim`（UNLOGGED 瘦身表），只保留聚合需要的 15 列。

### 4.3 建议增强

1. **物化视图替代**: 用 `MATERIALIZED VIEW` 替代手动 slim 表，支持 `REFRESH MATERIALIZED VIEW CONCURRENTLY`
2. **列式存储**: 如果未来数据量持续增长，考虑 Citus 列式压缩存储 或 TimescaleDB 的列式 hypertable
3. **索引策略**: 确认所有 JOIN 路径都有覆盖索引（`INCLUDE` 语法），避免回表

## 5. QA 断言覆盖率评估

### 5.1 已覆盖

| 类别 | 断言 | 覆盖 |
|------|------|------|
| 互斥性 | H∩E, H∩F, E∩F | ✅ |
| 守恒性 | Keep=H∪E∪F | ✅ |
| 幽灵检测 | HEF⊆Source | ✅ |
| 映射完整性 | Drop→Map | ✅ |
| H 准入 | 排除无效块 | ✅ |
| F 构造 | 排除 E 原子 | ✅ |
| 切分事件 | cnt=0 存在 | ✅ |
| Shard 覆盖 | Plan 完整 | ✅ |
| Per-Shard | 全覆盖 | ✅ |

### 5.2 未覆盖（建议补充）

| 类别 | 建议断言 | 语义 |
|------|---------|------|
| 数量合理性 | `h_members > 0` | H 不应为空 |
| 比例合理性 | `h_pct BETWEEN 10 AND 50` | H 占比应在合理范围 |
| 密度一致性 | `profile_pre.network_tier = profile_final.network_tier` 未出现全部降级 | 切分后不应全部降级 |
| 性能 | Step03 耗时 < 4h | 防止性能退化隐蔽 |
| 原子密度 | E 原子平均 density 在合理范围 | 防止阈值失效 |

## 6. 增量更新能力

### 6.1 当前状态

**完全不支持增量**。每次数据更新必须：
1. 生成新 `run_id`
2. 重跑 Phase 00-99 全链路
3. 预计耗时 6~14 小时

### 6.2 增量设计方向

```
增量数据到达
    │
    ├→ 受影响 Shard 识别（按 ip_long 范围定位）
    │
    ├→ 受影响 Natural Block 重建
    │     ├→ 行程变化? → 重新切分 → 重新画像
    │     └→ 行程不变? → 只更新成员计数
    │
    ├→ 受影响的 H/E/F 重新分类
    │
    └→ 增量 QA（只校验变化部分 + 全局守恒）
```

**关键挑战**:
- Natural Block 是"连续区间"定义，新增一个 IP 可能合并两个块或拆分一个块，影响下游全链路
- H 是全局定义（按 network_tier_final='中型网络'），增量需要全局视角

**建议**: 短期接受全量重跑，中期探索"分区增量"（仅重跑受影响 Shard），长期设计"增量引擎"。

## 7. 单点数据库

### 7.1 当前架构

单台 PostgreSQL（192.168.200.217），承载：
- 离线 Pipeline 写入（高 I/O）
- WebUI 实时查询
- 可能的在线 API 查询

### 7.2 风险

- 离线跑批时 WebUI 查询变慢
- 数据库故障无备份切换
- 读写混合导致 WAL 日志膨胀

### 7.3 建议

1. **读写分离**: 添加一台流复制从库，WebUI 和在线 API 走从库
2. **连接池**: PgBouncer，限制并发连接数
3. **定时快照**: `pg_dump` 或 `pg_basebackup` 定时备份
