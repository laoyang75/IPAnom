# IP库核心逻辑

这个目录是当前项目后续重构、数据库修正、协作交接的核心知识库。

设计目标：

- 即使其他临时审计材料丢失，也可以依靠这个目录重建主流程。
- 先固定业务逻辑，再约束实现，再做数据库修改，再做 UI 调整。
- 明确哪些是业务规则，哪些是实现策略，哪些是当前代码问题。

建议阅读顺序：

1. `01_业务方案版.md`
2. `02_实现规范版.md`
3. `03_核心表清单.md`
4. `04_切分算法清单.md`
5. `05_代码依据索引.md`
6. `06_数据库改造评估与方案.md`
7. `07_第一轮数据库修改清单.md`
8. `08_测试与服务器执行计划.md`

当前约束：

- 主流程必须围绕这条主线理解：
  - 统一基础数据
  - 识别候选对象
  - 纯化对象
  - 分流对象
  - 后续画像与标签
- 画像和标签是后处理层，不是主流程前置条件。
- 当前数据库部分优先于 UI 调整。

重构原则：

- 不能因为性能优化偷偷改变业务规则。
- 不能跳过中间层直接“简化成结果表”。
- 不能让标签反向定义 H/E/F。
- 不能让辅助修复脚本覆盖主流程定义。
