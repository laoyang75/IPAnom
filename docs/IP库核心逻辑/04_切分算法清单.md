# IP库核心逻辑：切分算法清单

这个文件只讲“已经确认存在于代码里的切分算法”。

目标：

- 把主流程中的切分规则和后处理中的切分规则分开。
- 把“业务规则”和“实现优化”分开。
- 让未来新增切分时，有一张清晰的对照表。

## 1. 主流程切分总览

```mermaid
flowchart TD
    A[连续地址段] --> B[候选筛选]
    B --> C[/64 边界窗口评估]
    C --> D[主切分触发器]
    D --> E[空洞强制切分]
    E --> F[形成 final block]
    F --> G[16-IP 二次密度切分]
```

## 2. 已确认主流程切分算法

### 算法 1：PreH 候选筛选

- 所在阶段：Step03 后处理
- 输入：
  - `profile_pre`
  - `block_natural`
- 当前主流程真实触发：
  - `keep_flag = true`
  - `valid_cnt > 0`
  - 跨 `/64` 边界
- 输出：
  - 写入 `preh_blocks`
- 作用：
  - 缩小需要切分的对象范围
- 归类：
  - 候选筛选
- 是否影响后续边界：
  - 会

### 算法 2：Report 跳变切分

- 所在阶段：Step04 主切分
- 输入：
  - `window_headtail_64`
  - 左右窗口的均值和波动
- 触发条件：
  - `ratio_report > 4`
  - `cvL < 1.1`
  - `cvR < 1.1`
- 输出：
  - 边界记为 `is_cut = true`
- 作用：
  - 切开两侧上报量级明显不同的对象
- 偏向：
  - 边界识别 + 对象纯化

### 算法 3：Mobile 跳变切分

- 所在阶段：Step04 主切分
- 输入：
  - 左右窗口移动设备统计
- 触发条件：
  - `mobile_diff > 0.5`
  - 或 `mobile_cnt_ratio > 4`
- 输出：
  - 边界记为 `is_cut = true`
- 作用：
  - 切开移动属性明显不同的对象
- 偏向：
  - 对象纯化

### 算法 4：Operator 切分

- 所在阶段：Step04 主切分
- 输入：
  - 左右窗口唯一运营商
- 触发条件：
  - 左右两侧运营商都存在
  - 且运营商不同
- 输出：
  - 边界记为 `is_cut = true`
- 作用：
  - 切开运营商归属明显不同的对象
- 偏向：
  - 边界识别

### 算法 5：Density 跳变切分

- 所在阶段：Step04 主切分
- 输入：
  - 左右窗口均设备密度
- 触发条件：
  - `ratio_devices > 10`
  - `cvL_dev < 1.5`
  - `cvR_dev < 1.5`
- 输出：
  - 边界记为 `is_cut = true`
- 作用：
  - 切开设备密度明显不同的对象
- 偏向：
  - 对象纯化

### 算法 6：Void Zone 强制切分

- 所在阶段：Step04 补充切分
- 输入：
  - `split_events_64`
  - 边界两侧 valid 计数
- 触发条件：
  - 连续大于 `2` 个 `/64` bucket 没有 valid IP
- 输出：
  - 空洞入口和出口强制标记 `is_cut = true`
- 作用：
  - 解决大空洞导致切分器失灵的问题
- 偏向：
  - 边界识别补丁
- 是否强制：
  - 是

### 算法 7：16-IP 二次密度切分

- 所在阶段：Step04 二次切分
- 输入：
  - 初步生成的 `block_final`
  - 块内成员设备数
- 触发条件：
  - 仅对长度 `>=64 IP` 的块
  - `16-IP` 窗口最大/最小均设备数比值 `>10`
  - 相邻窗口比值也 `>10`
- 输出：
  - 删除旧 final block
  - 写入新的子块
- 作用：
  - 修复 `/64` 内部仍存在的密度异质性
- 偏向：
  - 对象纯化
- 是否强制：
  - 满足条件即执行

## 3. 已确认的后处理切分算法

### 算法 8：E 超大段拆分

- 所在脚本：`split_oversized_e_runs.py`
- 所属层次：
  - E 的后处理
  - 不属于 H 主流程
- 输入：
  - `e_runs`
  - `e_cidr_summary`
  - `e_members`
- 触发条件：
  - `ip_count > 16384`
- 规则：
  - 优先在 B 类边界切开
  - 若子段仍超上限，再按 `16384` 硬切
- 输出：
  - 新 `e_run_id`
  - 更新 `e_members`
  - 重写 `e_runs`
  - 重建 `e_cidr_summary`
- 作用：
  - 控制 E 段过大导致的摘要和展示失真
- 归类：
  - 后处理治理

## 4. 切分的顺序与优先级

### 当前主流程顺序

1. PreH 候选筛选
2. `/64` 边界窗口统计
3. 主切分触发器并列计算
4. 空洞强制切分
5. 生成初版 final block
6. `16-IP` 二次密度切分

### 当前优先级特点

- 主切分触发器之间没有先后覆盖关系
- 它们是并列 OR 关系
- Void Zone 属于主切分后的补丁
- `16-IP` 属于最终块生成后的补充纯化

## 5. 已确认的强制切分

- Void Zone 强制切分
- E 超大段上限切分

## 6. 已确认的补充切分

- `16-IP` 二次密度切分

## 7. 当前没有实现的能力

这些不是“猜测缺失”，而是当前代码中没有看到明确实现：

- 最小块回退机制
- 切完过小后自动合并
- 切分优先级重算
- 切分后语义一致性评分

## 8. 新增切分算法的准入原则

未来如果增加新的切分算法，必须满足：

1. 目标是提高对象纯度，而不是单纯为了速度。
2. 必须明确它属于：
   - 候选筛选
   - 主切分
   - 强制切分
   - 补充切分
   - 后处理切分
3. 必须说明它会不会影响 H/E/F 边界。
4. 必须对比新增前后的：
   - H 块数
   - H 成员数
   - E/F 规模
   - 标签分布
5. 不能把实现层参数变化伪装成业务规则变化。
