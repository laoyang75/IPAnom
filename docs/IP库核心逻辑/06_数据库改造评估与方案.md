# IP库核心逻辑：数据库改造评估与方案

这份文档只讨论数据库逻辑，不讨论 UI 调整。

执行顺序固定为：

1. 先确认数据库改造方案
2. 再做小样本测试
3. 再做完整跑数
4. 最后再考虑 UI 调整

## 1. 当前数据库代码的主要问题

### 问题 1：Step03 存在两套 keep/drop 语义

现状：

- 标准 SQL `03_pre_profile_shard.sql` 已改成：
  - `keep_flag = true`
  - `ALL_ABNORMAL_BLOCK` 只做标记
- 但主编排实际调用的是 `orchestrate_step03_bucket_full.py`
  - 其中 `valid_cnt=0 -> keep_flag=false`

风险：

- 同一项目里，业务文档、静态 SQL、主编排实际跑法不一致
- 重跑后结果可能与当前理解完全不一样

建议：

- 先冻结 Step03 唯一语义
- 然后统一主编排、优化脚本、静态 SQL、文档

当前建议方向：

- 若以你现在的设计为准，建议保留异常 IP 标记，但不让异常值参与统计
- 至于“全异常块是否保留在主分类链路”，要单独定规则

### 问题 2：H 仍允许 `valid_cnt=0` 的大块进入

现状：

- `04P` 对 `valid_cnt=0` 的块仍会用 total fallback 评分
- `05` 只要求：
  - `network_tier_final` 属于 H 档
  - `member_cnt_total >= 4`
- 没有要求 `valid_cnt > 0`

结果：

- 最新 `sg_004` 里仍有 `161` 个 `valid_cnt=0` 的 H 块

风险：

- H 业务对象会混入“全异常但仍被评成大网络”的块
- 后续 H 摘要和标签无法稳定解释这些对象

建议：

- 先明确 H 是否允许 `valid_cnt=0`
- 如果不允许，应在 H 准入层显式加 `valid_cnt > 0`
- 如果允许，也必须在摘要层完整镜像并明确标记

### 问题 3：H 的小块问题没有从根上修，只是被入库门槛挡住

现状：

- `04` 切分仍会产生小块
- `04P` 小块仍可能得到高 tier
- `05` 通过 `member_cnt_total >= 4` 把小块挡在 H 外

风险：

- 一旦以后调整 H 门槛，问题会再次暴露
- 直接读 `profile_final` 的分析仍然会看到大量高 tier 小块

建议：

- 第一步先保住 H 门槛
- 第二步再评估是否在切分层增加最小业务块约束
- 这一步不要和主边界修复同时做，避免变量过多

### 问题 4：E 的 short_run 边界未统一

现状：

- 代码里 `run_len < 3` 只做 `short_run=true`
- 仍然进入 `e_members`

风险：

- E/F 边界不稳定
- 文档和 SQL 解释冲突

建议：

- 先冻结业务口径：
  - short_run 是否进 E
- 如果不进 E，应改 `07` 的 `e_members` 生成逻辑

### 问题 5：F 主流程和 F 修复脚本定义不一致

现状：

- 主流程 `08`：`F = R1 \\ E`
- 修复脚本 `rebuild_f_and_summary.py`：`F = source \\ H \\ E \\ drop`

风险：

- 重建 F 时可能与主流程不一致
- 摘要补建结果和主分类表可能脱节

建议：

- 明确主流程定义唯一以 `06 -> 07 -> 08` 为准
- 修复脚本必须改成与主流程同义

### 问题 6：摘要层与主分类层不完全对齐

现状：

- H 摘要会排除异常 IP 聚合，导致 `h_block_summary` 不能完整镜像 `h_blocks`
- E 摘要没有直接复用成员层的异常标记，异常计数暂时写 0
- F 摘要也直接从原始表取值

风险：

- 摘要不是主分类的稳定镜像
- UI 和标签看到的对象与主库对象不是同一层

建议：

- H/E/F 摘要统一基于主分类结果 + `source_members`
- 异常统计要统一从成员层继承，而不是各脚本自行处理

### 问题 7：shard 数处理仍有固定值遗留

现状：

- 主编排和很多脚本仍默认 `64`
- 数据库实际 run 已出现 `65` 和 `242`

风险：

- 补建摘要、后处理、重跑时容易缺 shard 或丢数据

建议：

- 全部改为从 `shard_plan` 或目标主表动态读取 shard 列表

## 3. 推荐的第一次修改范围

为了降低风险，第一次数据库改造不要同时动太多规则。

建议第一次只做这 4 项：

1. 统一 Step03 语义
2. 统一 H 准入规则
3. 冻结 E short_run 规则
4. 统一 F 主定义与修复脚本

暂时不要第一轮就做：

- 切分最小块回退
- 标签模型重构
- UI 查询逻辑

## 4. 推荐的修改方案骨架

### 方案 P0：冻结权威口径

先把下面这些写死到规范中：

- 异常 IP 是否打断连续性
- 全异常块是否允许进入 H/E/F
- H 是否必须 `member_cnt_total >= 4`
- E 是否接受 short_run
- F 的主定义是否以 `R1 \\ E` 为准

### 方案 P1：统一 Step03

目标：

- 主编排、优化脚本、静态 SQL 只有一套 keep/drop 语义

### 方案 P2：统一 H 准入

目标：

- H 的 SQL、文档、摘要构建语义一致

### 方案 P3：统一 E/F 边界

目标：

- short_run 语义和 F 收口语义一致

### 方案 P4：统一摘要层

目标：

- H/E/F 摘要全部以主分类表 + `source_members` 为准

### 方案 P5：统一动态 shard

目标：

- 所有补建和后处理脚本都不再写死 64

## 5. 测试方案

### 阶段 1：本地方案确认

- 先只输出修改 SQL / 脚本方案
- 不直接跑库

### 阶段 2：小样本测试

- 选 1 到 2 个 shard
- 验证：
  - H/E/F 守恒
  - H/E/F 互斥
  - H 边界是否符合新规则
  - E short_run 是否按新规则落位
  - 摘要层是否和主分类一致

### 阶段 3：完整跑数

- 在服务器上执行完整数据库重跑
- 先跑主分类
- 再跑摘要层
- 最后验收

### 阶段 4：UI 后置调整

- 只有数据库口径稳定后再改 UI
