# IP库核心逻辑：测试与服务器执行计划

这份文档描述后续数据库修改后的测试与执行顺序。

注意：

- 当前阶段只到“方案确认”
- 还没有进入服务器执行

## 1. 总执行顺序

固定顺序：

1. 确认修改方案
2. 本地整理补丁
3. 小样本数据库测试
4. 验证通过后再完整跑数
5. 数据库稳定后再处理 UI

## 2. 小样本测试目标

小样本测试的目标不是追求全量性能，而是确认主逻辑是否被改对。

至少验证：

1. Step03 语义是否统一
2. H 是否排除了 `valid_cnt=0`
3. H 是否排除了 `<4 IP` 小块
4. short_run 是否不再进入 E
5. F 是否与主流程定义一致
6. 守恒是否仍成立
7. H/E/F 是否仍然互斥

## 3. 建议的小样本范围

建议：

- 选择 `1-2` 个 shard
- 同时包含：
  - 有异常块
  - 有 H
  - 有 E
  - 有 F

优先原则：

- 不要选完全空 shard
- 不要只选单一结构 shard

## 5. 必须执行的验证 SQL

### 验证 1：H/E/F 互斥

```sql
SELECT COUNT(*) AS h_overlap_e
FROM rb20_v2_5.h_members h
JOIN rb20_v2_5.e_members e ON e.run_id=h.run_id AND e.ip_long=h.ip_long
WHERE h.run_id = :run_id;

SELECT COUNT(*) AS h_overlap_f
FROM rb20_v2_5.h_members h
JOIN rb20_v2_5.f_members f ON f.run_id=h.run_id AND f.ip_long=h.ip_long
WHERE h.run_id = :run_id;

SELECT COUNT(*) AS e_overlap_f
FROM rb20_v2_5.e_members e
JOIN rb20_v2_5.f_members f ON f.run_id=e.run_id AND f.ip_long=e.ip_long
WHERE e.run_id = :run_id;
```

目标：

- 全部为 `0`

### 验证 2：守恒

```sql
WITH keep_u AS (
  SELECT DISTINCT ip_long
  FROM rb20_v2_5.keep_members
  WHERE run_id = :run_id
),
hef_u AS (
  SELECT DISTINCT ip_long FROM rb20_v2_5.h_members WHERE run_id = :run_id
  UNION
  SELECT DISTINCT ip_long FROM rb20_v2_5.e_members WHERE run_id = :run_id
  UNION
  SELECT DISTINCT ip_long FROM rb20_v2_5.f_members WHERE run_id = :run_id
)
SELECT
  (SELECT COUNT(*) FROM keep_u) AS keep_cnt,
  (SELECT COUNT(*) FROM hef_u) AS hef_cnt;
```

目标：

- `keep_cnt = hef_cnt`

### 验证 3：H 不含全异常块

```sql
SELECT COUNT(*) AS valid0_h_blocks
FROM rb20_v2_5.h_blocks
WHERE run_id = :run_id AND valid_cnt = 0;
```

目标：

- `0`

### 验证 4：H 不含 `<4 IP` 小块

```sql
SELECT COUNT(*) AS h_lt4_blocks
FROM rb20_v2_5.h_blocks
WHERE run_id = :run_id AND member_cnt_total < 4;
```

目标：

- `0`

### 验证 5：short_run 不进入 E

```sql
SELECT COUNT(*) AS short_run_e_members
FROM rb20_v2_5.e_members em
JOIN rb20_v2_5.e_runs er
  ON er.run_id = em.run_id
 AND er.shard_id = em.shard_id
 AND er.e_run_id = em.e_run_id
WHERE em.run_id = :run_id
  AND er.short_run = true;
```

目标：

- `0`

### 验证 6：F 与主定义一致

```sql
SELECT COUNT(*) AS f_overlap_h
FROM rb20_v2_5.f_members f
JOIN rb20_v2_5.h_members h
  ON h.run_id = f.run_id AND h.ip_long = f.ip_long
WHERE f.run_id = :run_id;

SELECT COUNT(*) AS f_overlap_e
FROM rb20_v2_5.f_members f
JOIN rb20_v2_5.e_members e
  ON e.run_id = f.run_id AND e.ip_long = f.ip_long
WHERE f.run_id = :run_id;
```

目标：

- 都为 `0`

### 验证 7：摘要层与主分类层对齐

```sql
SELECT
  (SELECT COUNT(*) FROM rb20_v2_5.h_blocks WHERE run_id = :run_id) AS h_blocks,
  (SELECT COUNT(*) FROM rb20_v2_5.h_block_summary WHERE run_id = :run_id) AS h_summary,
  (SELECT COUNT(*) FROM rb20_v2_5.e_runs WHERE run_id = :run_id) AS e_runs,
  (SELECT COUNT(*) FROM rb20_v2_5.e_cidr_summary WHERE run_id = :run_id) AS e_summary,
  (SELECT COUNT(*) FROM rb20_v2_5.f_members WHERE run_id = :run_id) AS f_members,
  (SELECT COUNT(*) FROM rb20_v2_5.f_ip_summary WHERE run_id = :run_id) AS f_summary;
```

目标：

- H/E/F 摘要层与主分类层规模逻辑一致

## 6. 服务器执行计划

真正进入执行阶段时，顺序固定为：

### 阶段 A：本地补丁确认

- 在本地完成 SQL / Python 修改
- 先不直接跑服务器

### 阶段 B：SSH 到服务器

- 上传或同步补丁到服务器
- 在服务器上选小样本 shard 运行

### 阶段 C：验证 SQL

- 执行上述验证 SQL
- 确认边界、守恒、互斥、摘要对齐

### 阶段 D：完整重跑

- 小样本通过后，再跑全量

### 阶段 E：数据库冻结

- 确认数据库结果稳定
- 再开始 UI 调整
