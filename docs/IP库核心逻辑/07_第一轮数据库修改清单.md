# IP库核心逻辑：第一轮数据库修改清单

这份文档的目标是把“第一轮数据库修改”落到文件级别。

原则：

- 第一轮只修主分类边界和数据库一致性。
- 不做 UI 调整。
- 不同时引入过多变量。

## 1. 第一轮修改范围

第一轮只做 5 件事：

1. 统一 Step03 语义
2. 统一 H 准入规则
3. 冻结 E 的 short_run 规则
4. 统一 F 主定义和 F 修复脚本
5. 清理 shard 固定值遗留，改为动态读取

暂时不做：

- 最小块回退/重组
- 标签模型重构
- UI 查询逻辑修正

## 2. 当前采用的目标假设

如果没有额外新口径输入，第一轮默认按以下目标执行：

### 假设 A：异常 IP 不参与关键统计

- 异常 IP 保留在成员层
- 但不参与：
  - 连续段统计
  - 切分信号计算
  - 画像统计
  - 标签统计

### 假设 B：全异常块不进入业务主分类

- `valid_cnt = 0` 的块不进入 H/E/F
- 但保留在审计层

### 假设 C：H 仍保持最小块约束

- `member_cnt_total >= 4`

### 假设 D：E 不接收 short_run

- 第一轮默认目标：`run_len < 3` 不进入 E
- 如果你后续明确改成 `2`，只需要替换该规则值

### 假设 E：F 的主定义唯一以主流程为准

- `F = R1 \\ E`

## 3. 文件级修改清单

### 修改 1：统一 Step03 静态 SQL 与主编排语义

目标文件：

- `Y_IP_Codex_RB2_5/03_sql/RB20_03/03_pre_profile_shard.sql`

当前问题：

- 静态 SQL 里 `keep_flag = true`
- 主编排实际跑的是优化脚本，语义不同

目标改法：

- 把静态 SQL 的 `score/tier` 逻辑改成与优化版一致
- 具体包括：
  - `valid_cnt = 0` 时：
    - `wA = NULL`
    - `wD = NULL`
    - `simple_score = NULL`
    - `network_tier_pre = '无效块'`
    - `keep_flag = false`
    - `drop_reason = 'ALL_ABNORMAL_BLOCK'`
  - 不再用 `member_cnt_total` 和 `devices_sum_total` fallback 给全异常块评分

依据：

- 当前优化版 Step03 已是这个方向：
  - [orchestrate_step03_bucket_full.py](/Users/yangcongan/cursor/IP/Y_IP_Codex_RB2_5/04_runbook/orchestrate_step03_bucket_full.py#L300)
  - [orchestrate_step03_bucket_full.py](/Users/yangcongan/cursor/IP/Y_IP_Codex_RB2_5/04_runbook/orchestrate_step03_bucket_full.py#L309)

修改原因：

- 先消灭“一步两套口径”

### 修改 2：确认优化版 Step03 为主流程权威实现

目标文件：

- `Y_IP_Codex_RB2_5/04_runbook/orchestrate_step03_bucket_full.py`

当前问题：

- 语义虽然更接近目标，但还没有被正式写成“权威口径”

目标改法：

- 保留现有：
  - `valid_cnt = 0 -> keep_flag = false`
  - `network_tier_pre = '无效块'`
- 增加更明确注释，写清：
  - 全异常块不进入业务主分类
  - 异常 IP 只保留审计意义

修改原因：

- 让优化脚本不再只是“性能实现”，而是“主流程权威语义的一部分”

### 修改 3：统一 Step03 后处理

目标文件：

- `Y_IP_Codex_RB2_5/03_sql/RB20_03/03_post_process.sql`

当前问题：

- 当前 `preh_blocks` 已要求 `valid_cnt > 0`
- 但这层逻辑应和 Step03 权威语义一起固定

目标改法：

- 保留：
  - `keep_flag`
  - `valid_cnt > 0`
  - 跨 `/64`
- 增加注释说明：
  - `preh_blocks` 是切分候选
  - 全异常块不进入切分主流程

修改原因：

- 让候选筛选语义可重构

### 修改 4：统一最终评分语义，禁止全异常 final block 拿到高 tier

目标文件：

- `Y_IP_Codex_RB2_5/03_sql/RB20_04P/04P_final_profile_shard.sql`

当前问题：

- 当前 `04P` 对 `valid_cnt=0` 仍使用 total fallback
- 导致全异常 final block 仍可获得高 tier

目标改法：

- 当 `valid_cnt = 0` 时：
  - `density = NULL`
  - `report_density_valid = NULL`
  - `wA = NULL`
  - `wD = NULL`
  - `simple_score = NULL`
  - `network_tier_final = '无效块'`
- 只要 `valid_cnt > 0` 才参与正式评分

依据：

- 这和优化版 Step03 的思路保持一致

修改原因：

- 从根上禁止 `valid_cnt=0` 块被误打成大型/超大型网络

### 修改 5：收紧 H 准入

目标文件：

- `Y_IP_Codex_RB2_5/03_sql/RB20_05/05_h_blocks_and_members.sql`

当前问题：

- 当前已有限制 `member_cnt_total >= 4`
- 但没有显式 `valid_cnt > 0`

目标改法：

- 在 H 准入中额外增加：
  - `pf.valid_cnt > 0`

建议目标条件：

```sql
pf.network_tier_final IN ('中型网络','大型网络','超大网络')
AND pf.member_cnt_total >= 4
AND pf.valid_cnt > 0
```

修改原因：

- 双保险，防止无效块混入 H

### 修改 6：冻结 E 的 short_run 规则

目标文件：

- `Y_IP_Codex_RB2_5/03_sql/RB20_07/07_e_atoms_runs_members_shard.sql`

当前问题：

- 当前 short_run 只是标记，但仍进入 `e_members`

目标改法：

- 明确第一轮默认：
  - `run_len >= 3` 才进入 E
- 修改 `atom_to_run` 或 `e_members` 生成条件：
  - 只连接 `short_run = false` 的 `e_runs`

建议改法：

- 保留 `short_run` 字段，作为审计信息
- 但不让它进入 `e_members`

修改原因：

- 冻结 E/F 边界

### 修改 7：明确 08 是 F 的唯一主定义

目标文件：

- `Y_IP_Codex_RB2_5/03_sql/RB20_08/08_f_members_shard.sql`

当前问题：

- 08 本身问题不大，但要被明确成权威主定义

目标改法：

- 保持主逻辑不变：
  - `F = R1 \\ E`
- 增加更明确注释：
  - 这是 F 主流程真相
  - 后续修复脚本必须与此同义

修改原因：

- 后续所有 F 修复脚本都要回归它

### 修改 8：重写 F 修复脚本，使其与主流程同义

目标文件：

- `Y_IP_Codex_RB2_5/04_runbook/rebuild_f_and_summary.py`

当前问题：

- 当前脚本按 `source \\ H \\ E \\ Drop` 重建 F
- 与主流程定义不一致

目标改法：

- Step1 的 F 重建逻辑改为：
  - 从 `r1_members` 出发
  - 减去 E 覆盖
  - 与 `08_f_members_shard.sql` 同义

建议实现方式：

- 直接复用 `r1_members`
- 或直接复用 `08_f_members_shard.sql` 的 SQL 逻辑

修改原因：

- 防止补建脚本覆盖主流程定义

### 修改 9：修正 H 摘要层与主分类层的语义

目标文件：

- `Y_IP_Codex_RB2_5/04_runbook/build_h_block_summary.py`

当前问题：

- 当前脚本在聚合时排除了异常 IP
- 这会导致：
  - 真实 H 块存在
  - 但摘要层可能缺行

目标改法：

两种可选方案，第一轮推荐 A：

- A. 如果 H 主表已经禁止 `valid_cnt=0`，则继续保留“valid-only 聚合”，但必须确保每个 H 块仍然保留一行摘要
- B. 或直接统一为：
  - 统计字段区分 `valid` 与 `all`
  - 摘要层完整镜像 H 块

第一轮建议：

- 先配合 `05` 让 H 不再含 `valid_cnt=0`
- 再检查 `h_block_summary == h_blocks`

### 修改 10：修正 E 摘要层异常字段来源

目标文件：

- `Y_IP_Codex_RB2_5/04_runbook/build_e_cidr_summary.py`

当前问题：

- 当前 `abnormal_ip_count` / `abnormal_ip_ratio` 直接写 `0`

目标改法：

- 改为通过 `source_members` 继承：
  - `is_abnormal`
  - `is_valid`
- 让异常统计真实存在

修改原因：

- 摘要层必须继承成员层异常语义

### 修改 11：修正 F 摘要层来源

目标文件：

- `Y_IP_Codex_RB2_5/04_runbook/rebuild_f_and_summary.py`

当前问题：

- `f_ip_summary` 直接基于原始表
- 未统一继承成员层的异常/有效语义

目标改法：

- F 摘要以：
  - `f_members`
  - `source_members`
  - 原始字段
 共同构建

修改原因：

- F 摘要也要与主分类层统一

### 修改 12：所有补建脚本改成动态 shard

目标文件：

- `orchestrate_fresh_start_v2.py`
- `orchestrate_step03_bucket_full.py`
- `orchestrate_step11_chunked.py`
- `build_e_cidr_summary.py`
- `rebuild_f_and_summary.py`

当前问题：

- 多处仍默认 `64`
- 当前数据库已出现 `65` 和 `242` shard 的 run

目标改法：

- 统一改为：
  - 从 `shard_plan`
  - 或目标主表
 读取真实 shard 列表

重点改法：

- 主编排里不要再 `list(range(SHARD_CNT))`
- Step03 post process 不要再 `for sid in range(SHARD_CNT)`

修改原因：

- 避免丢 shard、缺数、错跑

## 4. 第一轮修改后的目标结果

第一轮改完以后，数据库应满足：

1. Step03 只有一套 keep/drop 语义
2. `valid_cnt=0` 的块不会进入 H
3. H 不含 `<4 IP` 小块
4. short_run 不再混进 E
5. F 修复脚本和主流程一致
6. H/E/F 摘要开始向主分类层收敛
7. 后处理脚本支持动态 shard

## 5. 第一轮暂不修改的内容

- UI 查询与文案
- 标签模型升级
- 最小块回退机制
- 切分触发阈值本身
- E 超大段拆分策略

## 6. 第一轮修改前的确认点

在真正动 SQL / 脚本之前，还需要你最终确认这 3 个口径：

1. `valid_cnt=0` 是否完全退出 H/E/F 业务主分类
2. E 的最小 run 长度是否固定为 `3`
3. F 的主定义是否唯一以 `R1 \\ E` 为准
