# PERFORMANCE_REPORT_RB20V2_5.md

## 1. 评估概述
本报告总结了 RB20 v2.5 在 `ip_loc2` 数据库环境下的性能表现。评估通过三个阶段（超小规模风险评估、500万规模 BENCH-1、5000万规模 BENCH-2）递进式验证，旨在识别 ShardPlan 逻辑中的性能爆炸点及死锁风险。

- **运行 ID**: `rb20v2_20260129_180000_bench_001` (5M), `rb20v2_20260129_180000_bench_002` (50M)
- **数据集核心参数**: 
  - 范围: [2,000,000,000, 2,050,000,000]
  - 归属国家: 中国

## 2. 核心指标对比 (Linearity Analysis)

| 指标 | Phase 2 (5M Range) | Phase 3 (50M Range) | 增长倍数 |
| :--- | :--- | :--- | :--- |
| **中国 IP 数量 (Source Members)** | 679,165 | 4,226,738 | **6.2x** |
| **自然块数量 (Natural Blocks)** | 206,370 | 1,027,739 | **5.0x** |
| **跨 Bucket 边界窗口数 (Window Rows)** | 6,991 | 45,524 | **6.5x** |
| **最终块数量 (Final Blocks)** | 206,550 | 1,028,666 | **5.0x** |

> [!NOTE]
> 从 5M 到 50M 的增长表现出高度的线性稳定性。Block 与 IP 的比例维持在 1:3 ~ 1:4 之间，符合大陆 IP 的分布特征。

## 3. 关键步骤风险评估

### RB20_02: Natural Block Identification
- **逻辑**: 使用 `SUM(is_new_block) OVER (ORDER BY ip_long)` 进行孤岛识别。
- **表现**: 在 420万行规模下，`SUM() OVER` 结合 `tmp_member_grp` 的创建非常平滑。
- **风险**: 随着全量数据（~8亿 IP）推入单 shard，此处可能是内存主要压力点。建议高并发下监控 `work_mem` 利用率。

### RB20_11: Window HeadTail Statistics
- **逻辑**: 对跨 bucket 边界进行 k=5 的前后探索。
- **表现**: 50M 范围产出 4.5万个窗口，处理速度极快。
- **结论**: 此前担心的“窗口逻辑爆炸”未发生。

### RB20_04: Split and Final Blocks
- **逻辑**: 基于窗口统计结果通过 15 条规则决定是否切割自然块。
- **表现**: 切割后的 Final Block 数量 (1,028,666) 仅比 Natural Block (1,027,739) 略微增加，说明切割频率较低，逻辑严密且无冗余。

## 4. 异常点与优化建议
1. **死锁风险**: 本次 Bench 采用单线程串行执行，未发现死锁。在全量 64 Shard 并发时，需确信 `rb20_v2_5` 下的各个 `DELETE/INSERT` 仅作用于特定 `run_id` 和 `shard_id`，以避免表级锁冲突。
2. **StepStats 幂等性**: 在测试过程中发现 `step_stats` 的 `PK` 冲突会导致事务回滚。
   - **优化**: 生产代码中已通过 `DELETE` 清理保证幂等，但在并发写 stats 时建议使用 `ON CONFLICT DO UPDATE`。

## 5. 结论
**RB20 v2.5 在 5000万规模（约全量 5%）的评估中表现优异，具备支撑全量生产的线性扩展性。**
建议下一步：进行 64 Shard 并发压力测试（sg_001）。
