# RB20 v2.0 性能评估（为 Y_IP_Codex_RB2_5 重跑做准备）

评估日期：2026-01-28  
评估 DB：`ip_loc2`  
评估对象（旧输出 schema）：`rb20_v2`（run_id=`rb20v2_20260107_194500_sg_001`）  
目标（新输出 schema）：`rb20_v2_5`

## 1) 结论（可行动）

1) **ShardPlan 的主要风险点是“全量排序 NTILE”**：对 ~5970 万中国成员做排序会显著拉长开跑时间并放大资源波动。  
   在 `Y_IP_Codex_RB2_5` 已改为 **/16 直方图近似分位**（避免全量排序），并在实测中单次扫描+聚合可在 ~20 秒级完成（见 2.1）。

2) **Step64（Window/切分评估）的核心计算在“小样本块集”上是毫秒~秒级**（见 2.2）。  
   说明“卡住”更可能来自：
   - 执行层漏跑 shard / 并发调度不严导致中断（历史 run 已发生）
   - 或者旧版 SQL 在 Join 顺序上退化（cand×map 重复回扫）导致不可预期的爆炸

## 2) 关键堵点实测（旧库上做基准）

### 2.1 ShardPlan（/16 直方图聚合）耗时

方法：对 `public."ip库构建项目_ip源表_20250811_20250824_v2_1"` 做 `ip_long >> 16` 分组计数（中国过滤），执行 `EXPLAIN (ANALYZE)`。

结果要点：
- 执行计划：Parallel Seq Scan + Sort + GroupAggregate
- **Execution Time：~19.6s**

解释：
- 仍然需要扫中国成员全量（不可避免），但不需要做 5970 万行的 `ORDER BY ip_long` 排序去算 `NTILE`。
- 因为 /16 的理论最大误差是 65536 行（单 /16 上限），对 `shard_cnt=64` 的均衡足够稳定。

### 2.2 Step64 Window 核心（5000 个 PreH 块小样本）

方法：在 `rb20_v2` 中选择一个已完整跑通的 shard（示例 shard_id=2），抽取 5000 个 `preh_blocks`，对其成员做 bucket64 维度窗口统计（包含 `ROW_NUMBER()` + top-5 聚合），执行 `EXPLAIN (ANALYZE)`。

结果要点：
- 样本规模：5000 个块，`m` 展开 valid 成员约 28942 行
- **Execution Time：~0.7s**

解释：
- 在“小样本”上非常稳定，说明核心算子（按 block/bucket 排序 + WindowAgg）并非必然瓶颈。
- 全量耗时主要由：
  - PreH 总块数
  - 每块成员分布
  - IO 命中率与并发度
  决定；因此更需要通过 `SHARD_CNT/CONCURRENCY` 严格控制并发并保证全覆盖。

## 3) 对 Y_IP_Codex_RB2_5 的落地要求（从旧库经验抽象）

1) **先 micro-bench 再全量**：先跑 `Y_IP_Codex_RB2_5/04_runbook/04_perf_eval_and_fix_tasks_v1.md` 的 B1/B2。  
2) **严格 shard_cnt 常量**：开跑前写死 `shard_cnt`，不得中途变更；最终必须跑 `RB20_99`（含 per-shard 全覆盖断言）。  
3) **执行入口统一**：优先用 `Y_IP_Codex_RB2_5/04_runbook/orchestrate_rb20_v2.py`，从 `shard_plan` 拉 shard 列表，避免手动 range 导致漏跑。
