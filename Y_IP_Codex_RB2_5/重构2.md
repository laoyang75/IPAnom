# Gemini 第一阶段（RB20）重构 v2.0 需求对齐文档（用于生成任务）

## 0. 文档定位与原则

### 0.1 文档用途

* 本文档是**下一步生成任务卡**的唯一需求口径：定义流程目标、输入锚点、核心规则、必须产物、执行纪律、验收标准。
* 本文档不承担“把每张表的完整字段表头一次写死”的工作；字段与统计的统一性通过**任务阶段的合同机制**完成（见第 6 节）。

### 0.2 设计原则

1. **实体库优先**：CIDR 块库必须是实体库；并且每一层关键阶段都要有实体库（块实体、原子实体、run 实体、切分事件实体）。
2. **真实成员口径**：最终成员集合 H/E/F 全部来自源表真实行（以 `ip_long` 为成员主键），禁止区间展开生成幽灵成员。
3. **并行可控**：依赖外部 orchestrator 多会话并行（因为 PG 自动并行不可控）；每个 per-shard 任务必须幂等可重跑。
4. **先对齐再执行**：任务阶段必须先产出“Schema/Metric/Report 合同”并人类确认，通过后才允许跑全量数据（见第 6 节）。

## 2. 全局口径（必须统一，所有统计与规则都依赖）

### 2.1 成员集合 vs valid 指标

* **成员集合（membership）**：包含异常 IP（异常只标记，不删除）。
* **valid 指标**：任何用于过滤、评分、密度、切分窗口、网络规模的指标，都必须剔除异常（`is_abnormal=false`）。
* **字段约束**：RB20_01 必须生成并固化 `is_abnormal` 与 `is_valid = NOT is_abnormal`；后续所有“valid 口径”均以 `is_valid=true` 为准。

> 说明：H/E/F 的“成员集合”默认均为 membership（即可包含 abnormal 成员），但其准入判定与画像/密度/切分窗口等指标统一使用 valid 口径（`is_valid=true`）。

### 2.2 “CIDR 块”在本项目中的定义

* 不是传统前缀 CIDR；本项目把 **连续整数区间（ip_long 相邻差 1）** 视为自然块（Natural Block）。
* “CIDR 块统计口径”：只把 **成员数（含异常）≥ 4** 的自然块计入“CIDR 块数”，≤3 只作为散点参考。

### 2.3 /27 原子定义（E/F 关键）

* `atom27_id = floor(ip_long / 32)`（一个原子对应 32 个 IP）
* 原子密度（valid 口径）：`atom_density = valid_ip_cnt / 32.0`，准入阈值：`atom_density >= 0.2`（等价 valid_ip_cnt>=7）
* 连续 run 最小长度：3；run_len < 3 仍保留为独立块并标记 short_run（便于审计）

### 2.4 禁止 BETWEEN / NOT BETWEEN

* 构建 F 必须用 `atom27_id` 等值 anti-join（或 range+GiST 备选），禁止用 `BETWEEN/NOT BETWEEN` 做范围反连接。

### 2.5 网络规模评估（画像）SIMPLE 口径（v2.0 固化，不可歧义）

> 本节定义 **网络规模评估函数**（SIMPLE / network_tier）的唯一口径：输入指标、分桶阈值、综合分与标签映射、边界条件处理。
> 该口径必须被写入 Metric Contract，并在 `network_tier_pre` 与 `network_tier_final` 两处**完全一致复用**（见 3.3、3.4、6.2）。

#### 2.5.1 输入指标（全部为 valid 口径）

对任意一个块实体（自然块或最终块），先定义其 **valid 成员集合**：

* `valid_members = { ip_long | ip_long 属于该块且 is_valid=true }`

定义以下输入指标：

1. **valid_cnt（块内 valid 成员数）**

* `valid_cnt = COUNT(DISTINCT ip_long) over valid_members`
* 说明：原则上 `ip_long` 在 Source Members 内应唯一；此处仍以 `DISTINCT` 作为口径兜底。

2. **devices_sum_valid（块内 valid 设备数求和）**

* `devices_sum_valid = SUM(设备数量) over valid_members`
* 字段来源：W 源表字段 `设备数量`（见 1.2）。
* NULL 处理：若 `设备数量` 存在 NULL，求和时按 SQL 默认语义处理（即忽略 NULL）；任务阶段需在 Metric Contract 明确是否需要 `COALESCE(设备数量,0)`，默认建议 `COALESCE` 为 0，以避免 NULL 导致低估。

3. **density（块内 valid 密度）**

* `density = devices_sum_valid / NULLIF(valid_cnt, 0)`
* 当 `valid_cnt=0` 时，`density` 必须为 **NULL**（不可为 0、不可为无穷大）。

#### 2.5.2 wA 分桶（按 valid_cnt）

当且仅当 `valid_cnt > 0` 时计算 `wA`；否则 `wA=NULL`（见 2.5.5）。

| valid_cnt 区间（闭区间） | wA |
| ----------------- | -: |
| 1 ~ 16            |  1 |
| 17 ~ 48           |  2 |
| 49 ~ 128          |  4 |
| 129 ~ 512         |  8 |
| ≥ 513             | 16 |

#### 2.5.3 wD 分桶（按 density）

当且仅当 `valid_cnt > 0`（从而 `density` 非 NULL）时计算 `wD`；否则 `wD=NULL`（见 2.5.5）。

| density 区间（边界不可歧义）  | wD |
| ------------------- | -: |
| density ≤ 3.5       |  1 |
| 3.5 < density ≤ 6.5 |  2 |
| 6.5 < density ≤ 30  |  4 |
| 30 < density ≤ 200  | 16 |
| density > 200       | 32 |

> 边界说明：`3.5` 归入第一档；`6.5` 归入第二档；`30` 归入第三档；`200` 归入第四档。

#### 2.5.4 综合分与网络规模标签

当且仅当 `wA` 与 `wD` 均非 NULL 时：

* `simple_score = wA + wD`

并由 `simple_score` 映射 `network_tier`（唯一枚举）：

| simple_score 条件（从高到低匹配） | network_tier |
| ----------------------- | ------------ |
| simple_score ≥ 40       | 超大网络         |
| 30 ≤ simple_score < 40  | 大型网络         |
| 20 ≤ simple_score < 30  | 中型网络         |
| 10 ≤ simple_score < 20  | 小型网络         |
| 0 ≤ simple_score < 10   | 微型网络         |

#### 2.5.5 valid_cnt=0（全异常块/无有效成员）处理规则（必须写死）

当 `valid_cnt = 0` 时，网络规模评估字段必须取如下固定规则：

* `devices_sum_valid = 0`（若按 NULL 忽略求和导致为 NULL，则在画像输出时必须标准化为 0；该标准化要求写入 Metric Contract）
* `density = NULL`
* `wA = NULL`
* `wD = NULL`
* `simple_score = NULL`
* `network_tier = '无效块'`（固定枚举值，不得用空字符串/NULL 代替）

并明确参与/不参与规则：

1. **不得参与 H 准入**：`network_tier='无效块'` 的块不得进入 H（见 3.5）。
2. **不得参与任何依赖 valid 指标的准入判定**：包括（但不限于）/27 原子密度准入、切分窗口指标判定、网络规模准入等；相关统计可保留用于审计，但不得用于准入与分层。
3. **实体保留用于审计**：`valid_cnt=0` 的自然块必须保留在自然块实体库与预画像实体库中，并在预画像中明确标记为 `Drop`（见 3.3），其成员映射必须存在（见 7.4）。

## 3. v2.0 的核心业务规则（必须固化）

## 3.1 RB20_01：过滤非中国 + 异常标记（只标记）

* 输入：W 源表 + A 异常表
* 输出：源成员初始库（中国范围）
* 规则：

  1. 非中国直接过滤掉（中国谓词由任务阶段确认后固化）
  2. 异常 IP 不删除，只标记 `is_abnormal=true`；并生成 `is_valid = NOT is_abnormal`

## 3.2 RB20_02：自然块识别（CIDR 自然块实体库）

* 输入：RB20_01 的中国成员（按 ip_long）
* 输出：

  * 自然块实体库（每行一个自然块）
  * 成员→自然块映射库（每行一个成员归属一个自然块）
* 规则：同 shard 内按 ip_long 排序，相邻差 1 则同块，否则断开。

> v2.0 要求：**CIDR 块库必须是实体库**，并且从 RB20_02 起就能用“块实体 + 成员映射”排障、对账。

## 3.3 RB20_03：预画像与 PreH 候选

* 输入：自然块实体库
* 输出：

  * 预画像实体库（包含 Keep/Drop、预估网络规模、drop_reason）
  * PreH 候选块实体库（用于后续切分范围裁剪）
* 注意：PreH 的定义属于 v2.0 的重构策略，目标是让切分只作用于更小集合，提升整体效率。

### 3.3.1 network_tier_pre（必须计算位置与口径锁死）

* **计算位置**：在 RB20_03（Pre Profile）阶段，对“自然块实体”计算 `network_tier_pre`。
* **唯一口径**：严格使用第 2.5 节 SIMPLE 口径，且必须为 valid 口径（`is_valid=true`）：

  * `valid_cnt`、`devices_sum_valid`、`density`、`wA`、`wD`、`simple_score`、`network_tier_pre`
* **一致性要求**：`network_tier_pre` 的阈值表与 `network_tier_final` 必须完全一致（见 3.4.1）。

### 3.3.2 valid_cnt=0 的预画像处理（必须写死）

当自然块 `valid_cnt=0` 时：

* 预画像必须输出 `network_tier_pre='无效块'`（按 2.5.5）
* 该块必须标记为 `Drop`
* `drop_reason` 固定写入：`ALL_ABNORMAL_BLOCK`（固定枚举值；如任务阶段需要英文/中文并存，可在 Config 合同给出映射，但该枚举语义不得变化）
* 该块不得进入 PreH 候选块实体库（即 PreH 必须排除 `valid_cnt=0`）

> 说明：Drop 并不影响“异常只标记不删除”；成员仍需保留在成员→块映射中用于审计与对账（见 7.4）。

## 3.4 Step64 切分（对 PreH 候选块进行切分）

### 3.4.1 候选切点与窗口

* 以 `bucket64 = floor(ip_long / 64)` 定义 64 桶边界；对每个块跨桶边界都评估切分点：`cut_ip_long = (i+1)*64`，i 遍历覆盖桶区间。
* 左右窗口各取 k=5 个**valid** IP（不足则 cnt<k）。
* 必须支持 cnt=0：指标为 NULL，默认不触发，但必须在 SplitEvents 记录。

### 3.4.2 三触发器（任一触发即切分）

* Report：`ratio_report > 4 AND cvL < 1.1 AND cvR < 1.1`
* Mobile：`mobile_diff > 0.5 OR mobile_cnt_ratio > 4`
* Operator：`opL IS NOT NULL AND opR IS NOT NULL AND opL <> opR`

### 3.4.3 多切点切分产出最终块实体

* 对同一个块收集所有触发的 cut_ip_long，升序切分成多个 segment。
* 最终块 ID 规则：`block_id_final = block_id_parent || '_' || lpad(segment_seq, 3, '0')`

### 3.4.4 性能要求（必须有“窗口摘要实体库”）

* 为避免对每个切点扫描整块，必须在切分前生成 block×bucket64 的 head/tail k=5 窗口摘要实体库。
* 窗口内运营商字段必须做“唯一运营商判定”（distinct=1 才写入，否则 NULL），避免混合窗口误触发。

## 3.5 H 库定义（中型网络）

* H 是“块级库”：以最终块画像中的 `network_tier_final='中型网络'` 作为准入条件。
* 不允许在 H 准入上叠加其它过滤（这是为了避免人为改变 H 的业务定义）。
* 补充约束：`network_tier_final='无效块'` 的块不得进入 H。

## 3.6 E/F 规则

* E：在 R1（剔除 H 覆盖成员后）按 /27 原子密度准入，再对连续原子合并成 run 块。
* F：`R1 - E_cov`，必须用 atom27 等值 anti-join，不造幽灵。

## 5. 执行与并行策略（必须满足）

### 5.1 run_id / shard_id

* 全部输出表必须带 `run_id`（隔离不同运行批次）。
* per-shard 步骤的输出必须带 `shard_id`（逻辑范围：`0..(shard_cnt-1)`），并支持按 shard 重跑。
  * `shard_cnt`：本次 run 的分片数量，默认 64；若遇到严重倾斜导致 64 分片无法完成，必须先走 DP-014 并在合同中写死。
  * 物理约束建议统一使用 `shard_id BETWEEN 0 AND 255`（上限 255），用 QA_Assert 强约束“实际 shard_id 必须是 `0..(shard_cnt-1)` 且连续”。
* 红线：**禁止执行中途修改 shard_cnt / 修改 shard_id 约束“补丁继续跑”**；若必须变更，只能新 run_id 重生成 ShardPlan 并重跑下游。

### 5.2 并行方式（PG 并行不可控时的硬策略）

* 采用 orchestrator 外部并发：一次并发运行 ≤32 个 shard（按 `shard_cnt` 分批/队列跑完）。
* 每个 per-shard 任务必须可单独执行、可重跑，不依赖其它 shard 的中间结果。

### 5.3 幂等与重跑纪律

* per-shard 输出：每次执行前必须清理 `run_id + shard_id` 的旧数据，再写入（防重复写入与回滚污染）。
* 全局输出：每次执行前必须清理 `run_id` 的旧数据，再写入。

### 5.4 ShardPlan（shard_cnt 分片）生成规则（v2.0 固化）

* 默认：`shard_cnt=64`；若严重倾斜需提高分片数，必须走 DP-014 并在合同里写死（禁止执行中途变更）。
* 生成策略（DP-012）：
  * **推荐（DP-012 选 C）**：按中国成员分布做 `NTILE(shard_cnt)` 分位切分，直接得到 `shard_cnt` 个近似均衡的 shard（避免“按全 IPv4 min/max 等分”在稀疏/成簇时产生空 shard）。
  * 备选：按 `ip_long` 范围等分 + 少量调整（仅在你明确要求保留字面实现时使用）。
* 若使用“范围等分 + 调整”的备选策略：基于统计结果做 **2 次调整**：

  * 数量少的 shard 放大范围
  * 数量大的 shard 缩小范围
  * 调整尺度按 int 范围的 **5%**
* 两次后停止，不再追求完全均衡；允许不均衡。
* 不做“断点对齐/连续性修复”，认为不会造成不可接受的不均衡。
* 失败策略：ShardPlan 生成失败或产生空 shard/重叠 shard，只能重生成 ShardPlan，并重跑受影响 shard 的下游步骤（不允许打补丁继续跑）。
* 必须断言（STOP）：
  * ShardPlan `count(*) = shard_cnt`，且 `shard_id` 必须连续覆盖 `0..(shard_cnt-1)`（无缺号/无重复）
  * `est_rows>0`（禁止空 shard）

### 5.5 终验收 Gate（STOP 即停机）

即使 H/E/F 汇总数字“看起来守恒”，也必须满足以下 Gate 才能验收：
* `qa_assert` 必须执行全套 STOP 断言（不允许只跑部分断言就宣布成功）。
* per-shard 关键实体必须对 ShardPlan 全覆盖（缺任一 shard 都不可验收）：
  * `source_members`
  * `map_member_block_natural`
  * 若 `preh_blocks>0`：必须有 `window_headtail_64`、`split_events_64`、`profile_final`
* 若出现严重倾斜/单 shard 失败：只能 STOP 并按 DP-014/DP-012 走“新 run_id + 新 ShardPlan + 重跑下游”的流程，不允许修改约束/临时加 shard 继续跑。

## 6. 任务阶段必须先完成的“合同机制”（确保表头与统计一致性）

这是你强调的关键点：**每一阶段的表头、统计方式必须由任务阶段 agent 通过 MCP 检查后与人类确认，并在所有步骤一致复用**。因此 v2.0 的任务生成必须包含以下“合同任务”，且通过后才能进入数据执行任务。

### 6.1 Schema Contract（表头合同）

**目标**：形成一份可执行的 DDL 方案与字段约束，覆盖所有关键库与必要索引策略（索引策略允许在执行任务里再评估，但字段与主键必须先统一）。

**任务产物（必须落文件/落表）**（本仓库已落地 v1 版本）：

* `Y_IP_Codex_RB2_5/02_contracts/schema_contract_v1.md`：表头/主键/必备字段/语义集合的唯一口径
* DDL/索引（幂等可重跑）：`Y_IP_Codex_RB2_5/03_sql/00_contracts/01_ddl_rb20_v2_full.sql`、`Y_IP_Codex_RB2_5/03_sql/00_contracts/02_indexes_rb20_v2.sql`

**必须的人类确认点**：

* 每张关键表“有哪些字段必须存在”（尤其所有计数与密度类数字字段不能漏）
* 各表的主键/唯一性约束（例如成员表 ip_long 是否唯一、映射表一对一/一对多）
* 网络规模评估相关字段的“语义字段集合”是否齐全（至少要能承载 2.5 节定义的 `valid_cnt/devices_sum_valid/density/wA/wD/simple_score/network_tier` 以及 pre/final 的 tier 字段）

### 6.2 Metric Contract（统计口径合同）

**目标**：把所有统计指标定义写成统一规范，避免每步各算各的导致对不上。

**必须固化的指标定义**（至少包含）：

* membership vs valid
* block 的 member_cnt_total / valid_cnt / abnormal_cnt / hole_cnt（hole 只算不补行）
* reports_sum_valid / devices_sum_valid / density
* Step64 的 ratio_report、cv、mobile_diff、mobile_cnt_ratio（含 cnt=0 的 NULL 规则）
* E 原子密度与 run 合并规则
* **网络规模评估（画像）SIMPLE 口径（必须写入合同且不可变）**：

  * 输入指标：`valid_cnt`、`devices_sum_valid`、`density`（全部 valid 口径，定义见 2.5.1）
  * 分桶阈值表：`wA`（按 valid_cnt）与 `wD`（按 density）（定义见 2.5.2/2.5.3）
  * `simple_score = wA + wD`（定义见 2.5.4）
  * `network_tier` 阈值表（定义见 2.5.4）
  * `valid_cnt=0` 时的固定规则：`density=NULL`、`simple_score=NULL`、`network_tier='无效块'`，以及 `devices_sum_valid` 的标准化要求（定义见 2.5.5）
* **tier_pre 与 tier_final 一致性要求（必须写入合同）**：

  * `network_tier_pre`：RB20_03 阶段对自然块按 2.5 口径计算
  * `network_tier_final`：RB20_04P 阶段对最终块按 **同一口径** 重新计算
  * 两者阈值表与边界必须一致，不允许“pre/final 两套口径”

**任务产物**（本仓库已落地 v1 版本）：

* `Y_IP_Codex_RB2_5/02_contracts/metric_contract_v1.md`（人类可读；版本锁定入口）

**必须的人类确认点**：

* “中国过滤谓词”与异常 join 规则（是否去重、空值处理）
* H/E/F 的集合口径（特别是 “成员来自真实行，禁止幽灵”）
* 网络规模评估字段映射（必须固化到 Config/Metric Contract）：

  * `devices_sum_valid` 使用 W 源表字段 `设备数量`（且仅对 `is_valid=true` 求和）
  * `valid_cnt` 来自块内 `ip_long` 计数（且仅计 `is_valid=true`）
  * `density` 使用 `devices_sum_valid / NULLIF(valid_cnt,0)`，并在 `valid_cnt=0` 时为 NULL

### 6.3 Report Contract（报告合同）

**目标**：每一步报告的表头与计算 SQL 统一，便于快速定位退化与漏跑 shard 的问题。

**必须产物**：

* 每一步都写 StepStats
* 每一步至少输出：

  1. 核心数表（成员数、块数）
  2. 变化解释表（规则命中与影响）

**任务产物**（本仓库已落地 v1 版本）：

* `Y_IP_Codex_RB2_5/02_contracts/report_contract_v1.md`
* 报告模板与可复现 SQL 锚点：`Y_IP_Codex_RB2_5/06_reports/01_gate1_sample_template.md`、`Y_IP_Codex_RB2_5/06_reports/02_gate2_full_pipeline_template.md`、`Y_IP_Codex_RB2_5/03_sql/`

**必须的人类确认点**：

* 报告表头、指标解释文字（中文说明）
* “块数口径”：CIDR 块统计只计 member_cnt_total>=4

## 7. 终验收（必须 STOP）

最终必须落 QA_Assert（severity=STOP），任一 FAIL 立即停止：

1. **互斥**：H/E/F 两两交集为 0（以 DISTINCT ip_long 为口径）
2. **守恒**：KeepMembers = H_cov ∪ E_cov ∪ F（真实 ip_long 集合）
3. **无幽灵**：H_cov ∪ E_cov ∪ F 必须是 Source Members 的子集（禁止区间展开造行）
4. **映射不蒸发**：Drop 的 member 也必须进入 Map_All（防 Drop 丢失导致对账失败）
5. **切分不退化**：SplitEvents 必须包含 cnt=0 记录；并且对 `valid_cnt>0` 的最终块，`network_tier_final` 不能单一退化（全空/全同）。同时必须单独统计 `network_tier_final='无效块'` 的块数量，用于审计（不得混进“正常 tier 分布”里误判）。

## 9. 本次 v2.0 已确定的“核心字段映射（用于规则计算）”

为了保证 Step64 与 H/E/F 能落地，本次环境下已知可用的关键字段名如下（均来自你提供的 schema dump）：

* `ip_long`、`ip_address`、`IP归属国家`、`IP归属运营商`、`上报次数`、`设备数量`、`移动网络设备数量`

* `density`：`devices_sum_valid / NULLIF(valid_cnt,0)`（理解与边界处理见 2.5）

1. **PreH 候选块的选择规则（当前主版本未写死）**
