# IP 库画像研究标准作业程序 (SOP)

> **上下文说明**: 本文档是我们进行 IP 库（`E库`、`H库`、`F库`）画像研究的**全局指导手册**。每次开始新的研究会话时，我（用户）会让你加载这个文档。当你（AI）看到这个文档时，你需要理解我们的整体宏观目标、当前的工作流，以及我们是如何在不同的子目录中隔离和推进具体研究的。

## 1. 研究目标与范围

我们的核心目标是对三个已划分的 IP 库进行深度的特征分析、边界测定和质量评估：
*   **H 库 (高置信度/大型网络)**: 约 1300万 IP。结构化程度高，主要为中大型网络块。
*   **E 库 (高密度原子)**: 约 4500万 IP。由高密度的 /27 网段和连续的 run 组成。
*   **F 库 (长尾碎片)**: 约 170万 IP。剩余的、未满足前两者条件的散点 IP。

**研究的最终交付物必须是可执行的、标准化的规则**，这些规则未来将被工程化，用于自动化的异常识别或分类优化。

## 2. 目录结构与上下文隔离

为了避免信息过载，我们将不同库的研究内容和发现隔离在特定的目录下。**在每次对话开始时，请根据我的指令，主动去加载对应目录下的文档以获取专属上下文：**

*   **`docs/08_profiling_E_database/`**: 存放所有关于 E 库的探查记录、图表设计要求、专属异常发现（如：“可疑的稀疏 run”）。
*   **`docs/09_profiling_H_database/`**: 存放所有关于 H 库的探查记录、内部聚类发现、以及与自然块/最终块的对比分析。
*   **`docs/10_profiling_F_database/`**: 存放所有关于 F 库的探查记录、特征归纳（如：是否包含被错漏的优质 IP，即“机会区”）。

## 3. 两阶段工作流

我们的研究工作分为两个紧密相连的阶段：

### 阶段一：数据准备、清理与规范化 (Phase 1)
在进行深层画像之前，必须保证地基牢固。
*   **目标**: 确保 H、E、F 库的数据符合既定 Pipeline (RB20 v2.5) 的逻辑，修正历史遗留的数据质量问题。
*   **活动**:
    *   验证数据一致性（如：验证守恒定律、检查库间重叠）。
    *   修复“已知问题”（例如：预画像时的 `wD` 分数跳跃过大、自然块判定时对 gap 容忍度为 0 导致碎片过多、HeadTail 窗口采样的 k=5 样本量太小等）。
    *   为可视化平台准备标准化的基础数据字段。
*   **注意**: 阶段一和阶段二不是绝对的一次性线性流程。**这是一个高度迭代的过程**，我们在阶段二（画像展示）中如果发现数据异常，会随时退回阶段一修改底层清洗逻辑。

### 阶段二：画像与可视化 (Phase 2)
这是研究的核心部分。
*   **目标**: 理解“是什么”构成了每个库，“为什么”IP会被分到这里，以及“如何”通过可视化手段把它们同库内的异常点区分开来。
*   **活动**:
    *   **单库宏观画像**: 分析单个库的全局统计特征。
    *   **边界分析**: 研究处于分类阈值边缘的 IP（比如：为什么这个 IP 掉进了 F 库而不是 E 库？）。
    *   **数据流转可视化**: 构建漏斗图或桑基图（Sankey），直观展示 5970万源 IP 是如何在 Pipeline 中一步步被过滤、分流到三个库的。
    *   **异常点探测可视**: 为特定库设计专用的图表（如散点图、箱线图、热力图），用于在海量数据中“点亮”异常（例如：定位到 E 库中表现异常的连续 run）。

## 4. 闭环迭代机制 (反馈循环)

研究是探索性的，你需要支持并适应这种工作方式：

1.  **可视化与发现**: 我们针对 E 库构建了一个新的可视化分析面。
2.  **观察到异常**: 在图表上，我们发现 E 库里聚集了一批特征非常散碎的 IP。
3.  **溯源根因**: 我们顺藤摸瓜，沿着 Pipeline 往前查，发现这个问题根源在于“阶段二 (Phase 02)”中自然块的 gap 容错率极度严苛。
4.  **回退至阶段一 (Phase 1)**: 我们暂停当前的画像设计，退回数据清理阶段。我们设计并测试一种新的 gap 容错逻辑。
5.  **重跑与验证**: 应用修复后，我们验证底层数据已修正。然后再次回到阶段二，确认新的可视化图表上，之前的异常碎片已经消失或被合理解释。

## 5. 对 Agent 的指导指令 (AI 行为准则)

当我们在执行本指导手册时，你（AI）应当遵守以下准则：

1.  **明确当前焦点**: 在对话中，总是要明确我们当前处于 **Phase 1 (修数据)** 还是 **Phase 2 (做画像)**，并且明确我们当前操作的是 **E、H、F 中的哪一个库**。
2.  **按需加载上下文**: 如果我说“我们今天研究 E 库”，你应该主动去读取 `docs/08_profiling_E_database/` 目录下的内容，而不是盲目猜测。
3.  **拥抱迭代**: 如果我在某个图表中指出了一个不合理的数据点，请主动帮我思考并列出：这个数据点是在 Pipeline 的哪一步产生的？我们该如何回溯并修复它？
4.  **聚焦规则沉淀**: 我们的每一次“发现”都必须转化为一条**假设**。被验证的假设必须转化为一条**标准化的逻辑规则**（例如：`IF density > X AND valid_cnt < Y THEN flag_as_anomaly`）。这就是我们最终要交付给工程团队的东西。
5.  **具备可视化思维**: 当你提出一个分析方案时，请用文字描述它在前端应该长什么样（比如：“这里适合用一个带有横向基准线的散点图，X轴是...，Y轴是...”），帮助我脑补最终的交互界面。
